BugONA-52372 ¬∑ In Review

App Crash ‚Äî Product Image Not Available

IndexOutOfBoundsException when selectedImageIndex = -1 passed to HorizontalPager

The Crash

The app crashed whenever a product had no image available. The log showed:

IndexOutOfBoundsException: Index -1, size 1
at androidx.compose.foundation.pager.PagerMeasureKt.measurePager

When no image was available, selectedImageIndex was set to -1. HorizontalPager expects a valid index between 0 and (size-1) ‚Äî passing -1 caused an immediate crash.

The Fix ‚Äî coerceIn()
val validatedIndex = selectedImageIndex.coerceIn(
0, (images.size - 1).coerceAtLeast(0)
)

// coerceIn clamps a value to a valid range:
// If value < min ‚Üí return min (0)
// If value > max ‚Üí return max
// Else ‚Üí return value unchanged

This guarantees a valid index is always passed to the pager regardless of image availability ‚Äî a reusable defensive pattern.

SecurityMEDIA-5010 ¬∑ Done ‚úì

POC ‚Äî Hardening API Keys Using Native .so Files (NDK)

Created a Proof of Concept to evaluate using Android NDK .so files to protect API key storage in APK/AAB builds

The Problem

API keys stored in Kotlin/Java code are easily exposed when an APK is decompiled. The Security Squad needed to evaluate whether moving secrets into native .so files (via NDK) would meaningfully improve resistance to reverse engineering.

What I Built
  • Created a standalone Android POC app/module using Android NDK
  • Stored a sample API key only in native code ‚Äî not in any Kotlin/Java file
  • Exposed the key to Kotlin via JNI (Java Native Interface)
  • Verified the key is accessible at runtime and not visible in decompiled Kotlin/Java
  • Tested behaviour in both debug and release builds
  • Decompiled the APK/AAB to analyse key visibility after native obfuscation
Findings & Recommendation
  • Security benefit: Key is not directly visible in decompiled Kotlin/Java ‚Äî a meaningful improvement over plain Kotlin storage
  • Limitation: Native code can still be reverse engineered using tools like Ghidra or IDA Pro, so it's not foolproof
  • Recommendation: NDK approach is worth pursuing as one layer of defence, especially when combined with encryption of the key itself
  • Full findings, comparison table, and next steps documented on Confluence

[Encryption] Best Practices for Storing Keys in Mobile Apps

Researched and documented best practices for securely storing keys and sensitive data in Android applications

Purpose

As part of the Security Squad's initiative to improve key management across the Tesco app, I was tasked with studying and documenting the current best practices for storing sensitive keys and data in mobile apps ‚Äî to inform the team's approach going forward.

What I Researched & Documented
  • EncryptedSharedPreferences ‚Äî Jetpack Security library for encrypting key-value stores on device
  • Android Keystore System ‚Äî hardware-backed key storage; keys never leave the secure enclave
  • NDK / .so files ‚Äî moving secrets to native code to reduce visibility in decompiled Java/Kotlin
  • Hardcoded keys in source ‚Äî documented why this is the worst approach and should be avoided
  • Comparison of all approaches: ease of implementation, security level, and attack surface
  • Clear recommendation on which approach to use for which type of secret
Impact

This research directly fed into ONA-53986, where I began implementing the NDK approach for the New Relic key in the Titan project. The Confluence document serves as the team's reference guide for key storage decisions.

[Android] Introduce .so File with 1 Key in Titan for Key Security

Applying the NDK approach in production ‚Äî storing the New Relic key of the android-titan project in a native .so file

What I'm Doing

Taking the learnings from the MEDIA-5010 POC and applying them directly to the production android-titan codebase. The first key being hardened is the New Relic API key ‚Äî moving it from plain Kotlin into a native .so file to protect it from APK decompilation.

Technical Approach
  • Introducing a .so file in the Titan project via Android NDK
  • New Relic key stored only in native C/C++ ‚Äî not visible in Kotlin/Java decompilation
  • Kotlin accesses the key at runtime via JNI bridge
  • Exploring XXTea encryption algorithm to add a second layer ‚Äî encrypting the key itself within the native code, so even native reverse engineering reveals only ciphertext
  • Investigating secure storage of the XXTea encryption/decryption key to avoid creating a new single point of failure
Why This Matters

This is real security work in a production codebase used by millions. Moving a key from plaintext Kotlin to encrypted native storage meaningfully raises the bar for attackers. The XXTea exploration shows I'm thinking beyond surface-level obfuscation to actual layered defence.
