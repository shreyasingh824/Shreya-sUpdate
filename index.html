Clean Architecture

The Tesco app follows Clean Architecture ‚Äî UI, Domain, and Data layers with strict inward-pointing dependencies. The Domain layer has zero knowledge of Android. This is the pattern I applied in my MVP2 side project and reinforce with every refactor I do.

MVVM Pattern

The Tesco app uses MVVM in the presentation layer. The ViewModel holds and exposes UI state; Views observe and react. This keeps UI logic testable and separated from business logic. My bug fixes for ONA-52372 (Pager crash) and ONA-52292 (UI clipping) were both diagnosed through this layered lens.

BugONA-52372 ¬∑ In Review

App Crash ‚Äî Product Image Not Available

IndexOutOfBoundsException when selectedImageIndex = -1 passed to HorizontalPager

The Crash

The app crashed whenever a product had no image available. The log showed:

IndexOutOfBoundsException: Index -1, size 1
at androidx.compose.foundation.pager.PagerMeasureKt.measurePager

When no image was available, selectedImageIndex was set to -1. HorizontalPager expects a valid index between 0 and (size-1) ‚Äî passing -1 caused an immediate crash.

The Fix ‚Äî coerceIn()
val validatedIndex = selectedImageIndex.coerceIn(
0, (images.size - 1).coerceAtLeast(0)
)

// coerceIn clamps a value to a valid range:
// If value < min ‚Üí return min (0)
// If value > max ‚Üí return max
// Else ‚Üí return value unchanged

This guarantees a valid index is always passed to the pager regardless of image availability ‚Äî a reusable defensive pattern.
